# Bennu Decision Diary

Record material design and implementation decisions here as they are made. Keep this diary append-only: add a new entry to correct or supersede an earlier decision instead of silently rewriting history.

## Entry format

### YYYY-MM-DD — Short decision title

- **Related issue:** Link or identifier
- **Context:** The problem, constraints, and relevant facts.
- **Decision:** What was selected, including its scope.
- **Alternatives considered:** Other credible options and why they were not selected.
- **Rationale:** Why this choice best fits the current constraints.
- **Consequences or follow-up:** Expected effects, risks, and any later work.
- **Validation/evidence:** Tests, measurements, review evidence, or other checks supporting the decision.

## Entries

### 2026-07-18 — Establish a minimal, normative contributor contract

- **Related issue/PR:** [Issue #1](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/1)
- **Context:** Bennu had no tracked files or implementation baseline. Before implementation begins, contributors need one repository-root contract that preserves the product direction, C++ architecture constraints, explicit error handling, scoped issue workflow, and ongoing decision record required by Issue #1. The issue explicitly excludes choices about syntax, execution model, C++ standard, build system, supported platforms, dependencies, and compiler or interpreter implementation.
- **Decision:** Bootstrap the repository with only root-level `AGENTS.md` and this diary. State the issue's contributor rules normatively and group them by product direction, C++ and architecture constraints, and engineering workflow. Treat Anka as linked design inspiration rather than a compatibility target. Use one append-only diary with a compact, reusable field-based entry format, and deliberately defer every listed non-goal.
- **Alternatives considered:** Copy Anka's architecture or behavior; add a build system or implementation skeleton; replace explicit constraints with generic style guidance; or create separate ADR files for each decision. These options were rejected because they would impose compatibility not requested by the issue, introduce speculative scaffolding, weaken discoverability and enforcement, or add unnecessary process for the project's current size.
- **Rationale:** Two focused Markdown files are the smallest complete bootstrap. A root `AGENTS.md` is automatically discoverable by contributors and agents, while a single diary makes decisions chronological and easy to inspect without selecting product or implementation details prematurely.
- **Consequences or follow-up:** Future contributions must follow `AGENTS.md`, document intentional differences from Anka, and append diary entries for material choices. Syntax, execution strategy, toolchain, platforms, dependencies, and implementation remain undecided until issue-driven work selects them. If a recorded decision changes, the correction must be a new entry.
- **Validation/evidence:** Both files were read back from the repository root and checked directly against every acceptance criterion in Issue #1. The complete untracked-file list contained only `AGENTS.md` and `doc/decision-diary.md`; inspection confirmed there is no compiler, interpreter, or unrelated scaffolding.

### 2026-07-18 — Adopt issue branches and local-main integration without pull requests

- **Related issue:** [Issue #3](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/3)
- **Context:** Bennu's contributor contract required issue-driven, scoped work but did not define how Coder isolates, integrates, or hands off an issue. The project needs a predictable path that gives Chef an already-integrated local result to verify while avoiding a GitHub pull-request review loop. A shared `main` also makes unconstrained concurrent integration unsafe.
- **Decision:** For every issue, Coder starts from a clean, current local `main`; creates one `issue/<number>-<short-slug>` branch; keeps scoped, issue-referencing commits on that branch; and runs all required validation there. After validation passes, Coder merges the branch into local `main`, pushes `main` to `origin`, leaves the GitHub issue open with exact branch, commit, merged-main, changed-file, push, and validation evidence, and blocks the matching Kanban card as `review-required`. Bennu does not use GitHub pull requests or GitHub code review. Unsafe integration stops and blocks with evidence. One active Coder issue is the default unless Chef explicitly establishes independent integration as safe.
- **Alternatives considered:** Use GitHub pull requests and reviews; commit issue work directly to `main`; leave validated issue branches unmerged for Chef; or allow multiple Coders to merge concurrently by default. Pull requests add a review mechanism the project deliberately does not want, direct commits remove issue-level isolation, unmerged branches shift integration work to verification, and default concurrency risks conflicts or remote divergence on the shared integration branch.
- **Rationale:** A short-lived issue branch provides isolation and auditable commits, while local merge plus a pushed `main` gives Chef one stable integrated range to inspect. Explicit evidence and stop conditions preserve independent verification without disguising integration as acceptance or forcing unsafe Git operations.
- **Consequences or follow-up:** Coder work is normally serialized. Local merge does not close the issue or accept the result; Chef still inspects the merged range, reruns tests, exercises the product, and either accepts the work or returns the same issue and card with exact unmet criteria. A dirty worktree, unexpected remote divergence, conflict, failed validation, or unrelated diff blocks integration until resolved.
- **Validation/evidence:** Issue #3 exercises this policy itself on `issue/3-local-main-workflow`. Validation reads both changed files from the repository root, checks the acceptance criteria and normative text, confirms only `AGENTS.md` and `doc/decision-diary.md` changed, inspects issue-referencing history and matching branch/main/origin SHAs, verifies fresh Hermes context discovery of root `AGENTS.md`, and confirms no pull request exists for the issue.

### 2026-07-18 — Establish the portable build, CLI contract, tests, and main CI

- **Related issue:** [Issue #4](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/4)
- **Context:** Bennu had no build configuration, executable, automated tests, CI, license, or clean-checkout instructions. This first runnable foundation must work across the Level 1 release platforms without introducing evaluator, backend, packaging, or speculative framework code. Anka currently builds through a Visual Studio 2022 solution with vcpkg and exposes `-r` and `-f`; Bennu needs a deliberately portable baseline and the named command surface selected by Issue #4 instead.
- **Decision:** Require CMake 3.20, Ninja, and C++20 without compiler extensions, and build one standard-library-only `bennu` executable from `src/main.cpp`. Generate a build-tree identity record containing the selected system, processor, compiler, and CMake version. Reserve `repl`, `run`, `emit-c`, and `build` as explicit subcommands that fail until later issues implement them. Use CTest plus a CMake script to execute the built binary and compare process status, stdout, and stderr for help, missing arguments, unknown input, and every reserved command; this creates no separate test executable or runtime source-tree dependency. Run the same Release configure, build, and CTest commands on pushes to `main` and manual dispatches on fixed Linux x64, Windows x64, and macOS arm64 runner labels. Each job records its observed OS and CPU architecture, fails on a matrix mismatch, and reports compiler, CMake, and Ninja identities. Pin the checkout and Windows MSVC-environment actions to commits.
- **Alternatives considered:** Copy Anka's Visual Studio solution and vcpkg baseline; use compiler-specific project files; add a third-party CLI or test framework; use shell-specific test scripts; implement the reserved commands now; or run CI for pull requests. These options were rejected because they reduce portability, add dependencies or runtime behavior outside this issue, make process assertions less uniform, or contradict Bennu's local-main workflow.
- **Rationale:** One executable, one small process-test driver, and one three-runner CI matrix are the smallest complete foundation. Explicit C++ standard requirements and pinned, versioned runner inputs make costs and platform behavior visible. Recognizing reserved commands without implementing them stabilizes the user-facing shape while keeping the language entirely absent.
- **Consequences or follow-up:** Later issues can replace each reserved failure with real behavior while retaining its process contract. The CLI currently performs no source I/O and needs no files beside the executable. The integrated `main` CI run remains the required cross-platform gate; runner-label or image retirement will require a new diary entry and workflow update.
- **Validation/evidence:** During test-driven implementation, each of the eight process contracts was observed failing before its matching behavior and then passing in `/tmp/bennu-tdd-build`. A clean GNU 15.2.0 Release configure/build passed all 8 CTest cases with `-std=c++20`; the generated identity recorded Linux x86_64, GNU 15.2.0, and CMake 4.4.0. The copied executable passed help and all failure contracts from outside the source tree, a warning-clean compile passed, and actionlint 1.7.12 accepted the workflow. The integrated three-platform GitHub Actions run remains required before Issue #4 handoff.

### 2026-07-18 — Use flat Level 1 programs, explicit results, and bounded arrays

- **Related issue:** [Issue #5](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/5)
- **Context:** The first language core must provide one reusable source-to-value path for decimal integers, `inc`, and `ioata` without coupling semantics to the CLI or later backends. It must preserve useful source locations, reject signed overflow and malformed input explicitly, and prevent a small untrusted request from causing an uncontrolled array allocation. Anka is semantic inspiration, but its broader grammar and array lifting are outside Level 1.
- **Decision:** Represent Level 1 integers with `std::int64_t` and convert literals with non-throwing `std::from_chars`. Represent a value as a public tag plus scalar field and contiguous `std::vector<std::int64_t>` array field. Tokens store byte offsets, one-based lines and columns, lengths, and whether horizontal whitespace preceded them. Parse each nonblank line into one contiguous postorder node array plus source-ordered root indices; evaluate that array in one forward pass rather than building an owning pointer tree or recursively evaluating calls. Expose tokenization, parsing, primitive application, complete-program evaluation, one-expression evaluation, and formatting as free functions with boolean result records carrying structured errors. Limit `ioata` to 1,000,000 elements, checked before `reserve`, and apply the same 1,000,000-element cumulative budget to all `ioata` results retained by one program evaluation. `ioata` of a nonpositive integer produces an empty vector without reserving, and `inc` rejects arrays rather than lifting over them.
- **Alternatives considered:** Use machine-width integers; use exceptions or add an expected/variant dependency; represent each value as a polymorphic object; build a recursive owning syntax tree; recursively evaluate nested calls; allow each program line its own independent million-element budget; or copy Anka's `inc` array lifting and broader syntax. These options were rejected because they weaken cross-platform semantics, violate explicit-error or data/behavior separation constraints, add dependencies or pointer ownership, make deep prefix programs consume the call stack, permit large amplification across a small multi-line source, or expand the specified language.
- **Rationale:** The tagged value keeps scalar access direct and array storage contiguous. Postorder nodes make every argument an earlier index, so ownership and evaluation order remain visible and nested calls have predictable linear traversal. A million signed 64-bit elements has an 8,000,000-byte (about 7.63 MiB) raw payload; checking both each request and the retained program total before allocation gives a deterministic, testable bound while remaining large enough for Level 1 examples. Explicit result records keep all expected failures in ordinary control flow and make line-and-column reporting available to later presentation layers.
- **Consequences or follow-up:** The inactive field in each tagged value is intentional small structural overhead. Tokens refer to source ranges by offset, so callers pass the same source buffer to parsing but tokens do not retain borrowed string views. Program evaluation retains intermediate values until the forward pass finishes; the cumulative array budget bounds that storage. Columns count source bytes/code units, which is sufficient because Level 1 tokens are ASCII. Later CLI or backend issues should consume this API rather than duplicate grammar or semantics, and any future Unicode, wider value model, lifting, or different resource policy requires a new language decision.
- **Validation/evidence:** Test-first cycles observed missing API or behavior failures before scalar evaluation, signed boundaries, `inc`, `ioata`, formatting, complete-program ordering, token locations, flat parsing, direct primitive errors, and the cumulative allocation guard were implemented. The focused evaluator test then passed with exact scalar/array formatting, nested calls, LF/CRLF/blank-line handling, missing final newline, every required malformed/type/range/allocation error class, exact locations, and both per-request and cumulative 1,000,000-element boundaries. A clean GNU 15.2.0 Release build passed all 9 CTest cases, the evaluator case passed under combined AddressSanitizer and UndefinedBehaviorSanitizer instrumentation, and a `-Wall -Wextra -Wpedantic -Wconversion -Wsign-conversion -Werror` build was clean. Integrated Linux, Windows, and macOS CI remains required before handoff.

### 2026-07-18 — Validate public evaluator data before transformation

- **Related issue:** [Issue #5](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/5)
- **Context:** The Level 1 API deliberately exposes plain token and expression structs. Callers can therefore construct values that the tokenizer and parser would never emit. Review found that unchecked token spans could overflow before substring access, unknown expression tags could select the wrong primitive path, and incomplete token streams could silently ignore source bytes or report misleading end locations.
- **Decision:** Treat every caller-supplied `Token` and `Program` as untrusted structural input. Validate token spans with subtraction after proving the offset is in range, reject unknown token and expression tags, require exactly one final zero-length end token at `source.size()`, and reject tokens following any earlier end token. When an end token is missing, derive its error location by scanning the source to the actual byte, line, and column at EOF rather than trusting the last supplied token.
- **Alternatives considered:** Document that only tokenizer output may be parsed; use `offset + length` range checks; let unknown enum tags fall through existing branches; accept an end token anywhere in the source; or report the last token location for a missing end marker. These options were rejected because public data remains constructible, addition can wrap, fallthrough changes semantics, an early end can hide source text, and the last token start is not the source end.
- **Rationale:** One linear validation pass keeps failure behavior explicit and bounded before parsing touches source ranges or evaluates public nodes. Requiring the end marker to describe EOF preserves the parser's complete-buffer contract, while source-derived EOF locations make diagnostics deterministic even when the token vector is empty or truncated.
- **Consequences or follow-up:** Parsing public tokens performs an additional linear validation pass and an EOF scan only on the missing-end error path. Token line and column fields remain caller-supplied diagnostic metadata; this change validates memory safety, known tags, and stream termination without duplicating tokenization or adding a second token representation.
- **Validation/evidence:** Focused regressions were observed failing before the corresponding corrections for overflow-sized spans, unknown expression tags, nonzero-length and early end tokens, and missing-end location handling. The evaluator test now covers malformed public spans and tags, empty and truncated token streams, tokens after an early end, a final end before source EOF, nonzero-length end tokens, and exact LF/CRLF-aware multiline EOF locations. Full Release, strict no-exceptions/warnings, sanitizer, adversarial, and integrated CI gates remain required before handoff.
