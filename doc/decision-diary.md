# Bennu Decision Diary

Record material design and implementation decisions here as they are made. Keep this diary append-only: add a new entry to correct or supersede an earlier decision instead of silently rewriting history.

## Entry format

### YYYY-MM-DD — Short decision title

- **Related issue:** Link or identifier
- **Context:** The problem, constraints, and relevant facts.
- **Decision:** What was selected, including its scope.
- **Alternatives considered:** Other credible options and why they were not selected.
- **Rationale:** Why this choice best fits the current constraints.
- **Consequences or follow-up:** Expected effects, risks, and any later work.
- **Validation/evidence:** Tests, measurements, review evidence, or other checks supporting the decision.

### 2026-07-18 — Adopt issue branches and local-main integration without pull requests

- **Related issue:** [Issue #3](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/3)
- **Context:** Bennu's contributor contract required issue-driven, scoped work but did not define how Coder isolates, integrates, or hands off an issue. The project needs a predictable path that gives Chef an already-integrated local result to verify while avoiding a GitHub pull-request review loop. A shared `main` also makes unconstrained concurrent integration unsafe.
- **Decision:** For every issue, Coder starts from a clean, current local `main`; creates one `issue/<number>-<short-slug>` branch; keeps scoped, issue-referencing commits on that branch; and runs all required validation there. After validation passes, Coder merges the branch into local `main`, pushes `main` to `origin`, leaves the GitHub issue open with exact branch, commit, merged-main, changed-file, push, and validation evidence, and blocks the matching Kanban card as `review-required`. Bennu does not use GitHub pull requests or GitHub code review. Unsafe integration stops and blocks with evidence. One active Coder issue is the default unless Chef explicitly establishes independent integration as safe.
- **Alternatives considered:** Use GitHub pull requests and reviews; commit issue work directly to `main`; leave validated issue branches unmerged for Chef; or allow multiple Coders to merge concurrently by default. Pull requests add a review mechanism the project deliberately does not want, direct commits remove issue-level isolation, unmerged branches shift integration work to verification, and default concurrency risks conflicts or remote divergence on the shared integration branch.
- **Rationale:** A short-lived issue branch provides isolation and auditable commits, while local merge plus a pushed `main` gives Chef one stable integrated range to inspect. Explicit evidence and stop conditions preserve independent verification without disguising integration as acceptance or forcing unsafe Git operations.
- **Consequences or follow-up:** Coder work is normally serialized. Local merge does not close the issue or accept the result; Chef still inspects the merged range, reruns tests, exercises the product, and either accepts the work or returns the same issue and card with exact unmet criteria. A dirty worktree, unexpected remote divergence, conflict, failed validation, or unrelated diff blocks integration until resolved.
- **Validation/evidence:** Issue #3 exercises this policy itself on `issue/3-local-main-workflow`. Validation reads both changed files from the repository root, checks the acceptance criteria and normative text, confirms only `AGENTS.md` and `doc/decision-diary.md` changed, inspects issue-referencing history and matching branch/main/origin SHAs, verifies fresh Hermes context discovery of root `AGENTS.md`, and confirms no pull request exists for the issue.

### 2026-07-18 — Establish the portable build, CLI contract, tests, and main CI

- **Related issue:** [Issue #4](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/4)
- **Context:** Bennu had no build configuration, executable, automated tests, CI, license, or clean-checkout instructions. This first runnable foundation must work across the Level 1 release platforms without introducing evaluator, backend, packaging, or speculative framework code. Anka currently builds through a Visual Studio 2022 solution with vcpkg and exposes `-r` and `-f`; Bennu needs a deliberately portable baseline and the named command surface selected by Issue #4 instead.
- **Decision:** Require CMake 3.20, Ninja, and C++20 without compiler extensions, and build one standard-library-only `bennu` executable from `src/main.cpp`. Generate a build-tree identity record containing the selected system, processor, compiler, and CMake version. Reserve `repl`, `run`, `emit-c`, and `build` as explicit subcommands that fail until later issues implement them. Use CTest plus a CMake script to execute the built binary and compare process status, stdout, and stderr for help, missing arguments, unknown input, and every reserved command; this creates no separate test executable or runtime source-tree dependency. Run the same Release configure, build, and CTest commands on pushes to `main` and manual dispatches on fixed Linux x64, Windows x64, and macOS arm64 runner labels. Each job records its observed OS and CPU architecture, fails on a matrix mismatch, and reports compiler, CMake, and Ninja identities. Pin the checkout and Windows MSVC-environment actions to commits.
- **Alternatives considered:** Copy Anka's Visual Studio solution and vcpkg baseline; use compiler-specific project files; add a third-party CLI or test framework; use shell-specific test scripts; implement the reserved commands now; or run CI for pull requests. These options were rejected because they reduce portability, add dependencies or runtime behavior outside this issue, make process assertions less uniform, or contradict Bennu's local-main workflow.
- **Rationale:** One executable, one small process-test driver, and one three-runner CI matrix are the smallest complete foundation. Explicit C++ standard requirements and pinned, versioned runner inputs make costs and platform behavior visible. Recognizing reserved commands without implementing them stabilizes the user-facing shape while keeping the language entirely absent.
- **Consequences or follow-up:** Later issues can replace each reserved failure with real behavior while retaining its process contract. The CLI currently performs no source I/O and needs no files beside the executable. The integrated `main` CI run remains the required cross-platform gate; runner-label or image retirement will require a new diary entry and workflow update.
- **Validation/evidence:** During test-driven implementation, each of the eight process contracts was observed failing before its matching behavior and then passing in `/tmp/bennu-tdd-build`. A clean GNU 15.2.0 Release configure/build passed all 8 CTest cases with `-std=c++20`; the generated identity recorded Linux x86_64, GNU 15.2.0, and CMake 4.4.0. The copied executable passed help and all failure contracts from outside the source tree, a warning-clean compile passed, and actionlint 1.7.12 accepted the workflow. The integrated three-platform GitHub Actions run remains required before Issue #4 handoff.

### 2026-07-18 — Use flat Level 1 programs, explicit results, and bounded arrays

- **Related issue:** [Issue #5](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/5)
- **Context:** The first language core must provide one reusable source-to-value path for decimal integers, `inc`, and `ioata` without coupling semantics to the CLI or later backends. It must preserve useful source locations, reject signed overflow and malformed input explicitly, and prevent a small untrusted request from causing an uncontrolled array allocation. Anka is semantic inspiration, but its broader grammar and array lifting are outside Level 1.
- **Decision:** Represent Level 1 integers with `std::int64_t` and convert literals with non-throwing `std::from_chars`. Represent a value as a public tag plus scalar field and contiguous `std::vector<std::int64_t>` array field. Tokens store byte offsets, one-based lines and columns, lengths, and whether horizontal whitespace preceded them. Parse each nonblank line into one contiguous postorder node array plus source-ordered root indices; evaluate that array in one forward pass rather than building an owning pointer tree or recursively evaluating calls. Expose tokenization, parsing, primitive application, complete-program evaluation, one-expression evaluation, and formatting as free functions with boolean result records carrying structured errors. Limit `ioata` to 1,000,000 elements, checked before `reserve`, and apply the same 1,000,000-element cumulative budget to all `ioata` results retained by one program evaluation. `ioata` of a nonpositive integer produces an empty vector without reserving, and `inc` rejects arrays rather than lifting over them.
- **Alternatives considered:** Use machine-width integers; use exceptions or add an expected/variant dependency; represent each value as a polymorphic object; build a recursive owning syntax tree; recursively evaluate nested calls; allow each program line its own independent million-element budget; or copy Anka's `inc` array lifting and broader syntax. These options were rejected because they weaken cross-platform semantics, violate explicit-error or data/behavior separation constraints, add dependencies or pointer ownership, make deep prefix programs consume the call stack, permit large amplification across a small multi-line source, or expand the specified language.
- **Rationale:** The tagged value keeps scalar access direct and array storage contiguous. Postorder nodes make every argument an earlier index, so ownership and evaluation order remain visible and nested calls have predictable linear traversal. A million signed 64-bit elements has an 8,000,000-byte (about 7.63 MiB) raw payload; checking both each request and the retained program total before allocation gives a deterministic, testable bound while remaining large enough for Level 1 examples. Explicit result records keep all expected failures in ordinary control flow and make line-and-column reporting available to later presentation layers.
- **Consequences or follow-up:** The inactive field in each tagged value is intentional small structural overhead. Tokens refer to source ranges by offset, so callers pass the same source buffer to parsing but tokens do not retain borrowed string views. Program evaluation retains intermediate values until the forward pass finishes; the cumulative array budget bounds that storage. Columns count source bytes/code units, which is sufficient because Level 1 tokens are ASCII. Later CLI or backend issues should consume this API rather than duplicate grammar or semantics, and any future Unicode, wider value model, lifting, or different resource policy requires a new language decision.
- **Validation/evidence:** Test-first cycles observed missing API or behavior failures before scalar evaluation, signed boundaries, `inc`, `ioata`, formatting, complete-program ordering, token locations, flat parsing, direct primitive errors, and the cumulative allocation guard were implemented. The focused evaluator test then passed with exact scalar/array formatting, nested calls, LF/CRLF/blank-line handling, missing final newline, every required malformed/type/range/allocation error class, exact locations, and both per-request and cumulative 1,000,000-element boundaries. A clean GNU 15.2.0 Release build passed all 9 CTest cases, the evaluator case passed under combined AddressSanitizer and UndefinedBehaviorSanitizer instrumentation, and a `-Wall -Wextra -Wpedantic -Wconversion -Wsign-conversion -Werror` build was clean. Integrated Linux, Windows, and macOS CI remains required before handoff.

### 2026-07-18 — Validate public evaluator data before transformation

- **Related issue:** [Issue #5](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/5)
- **Context:** The Level 1 API deliberately exposes plain token and expression structs. Callers can therefore construct values that the tokenizer and parser would never emit. Review found that unchecked token spans could overflow before substring access, unknown expression tags could select the wrong primitive path, and incomplete token streams could silently ignore source bytes or report misleading end locations.
- **Decision:** Treat every caller-supplied `Token` and `Program` as untrusted structural input. Validate token spans with subtraction after proving the offset is in range, reject unknown token and expression tags, require exactly one final zero-length end token at `source.size()`, and reject tokens following any earlier end token. When an end token is missing, derive its error location by scanning the source to the actual byte, line, and column at EOF rather than trusting the last supplied token.
- **Alternatives considered:** Document that only tokenizer output may be parsed; use `offset + length` range checks; let unknown enum tags fall through existing branches; accept an end token anywhere in the source; or report the last token location for a missing end marker. These options were rejected because public data remains constructible, addition can wrap, fallthrough changes semantics, an early end can hide source text, and the last token start is not the source end.
- **Rationale:** One linear validation pass keeps failure behavior explicit and bounded before parsing touches source ranges or evaluates public nodes. Requiring the end marker to describe EOF preserves the parser's complete-buffer contract, while source-derived EOF locations make diagnostics deterministic even when the token vector is empty or truncated.
- **Consequences or follow-up:** Parsing public tokens performs an additional linear validation pass and an EOF scan only on the missing-end error path. Token line and column fields remain caller-supplied diagnostic metadata; this change validates memory safety, known tags, and stream termination without duplicating tokenization or adding a second token representation.
- **Validation/evidence:** Focused regressions were observed failing before the corresponding corrections for overflow-sized spans, unknown expression tags, nonzero-length and early end tokens, and missing-end location handling. The evaluator test now covers malformed public spans and tags, empty and truncated token streams, tokens after an early end, a final end before source EOF, nonzero-length end tokens, and exact LF/CRLF-aware multiline EOF locations. Full Release, strict no-exceptions/warnings, sanitizer, adversarial, and integrated CI gates remain required before handoff.

### 2026-07-18 — Require exact token and source correspondence

- **Related issue:** [Issue #5](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/5)
- **Context:** The prior public-data validation checked token bounds, known tags, and final end-marker placement independently, but it did not prove that the ordered tokens described the supplied complete source. A caller could omit non-whitespace bytes, overlap or reorder spans, forge a primitive kind for a different lexeme, or supply false whitespace and location metadata. The parser could then accept and evaluate semantics that were not present in the source buffer.
- **Decision:** Retain the public `parse_program(source, tokens)` staged API, but treat `tokenize(source)` as the canonical complete lexical description. After the existing overflow-safe structural checks and before building any expression nodes, tokenize the source independently and require the supplied vector to match the canonical vector exactly in count, order, kind, byte offset, byte length, line, column, and separation flag. Reject every mismatch as a located `invalid_program` error; if the source itself cannot tokenize, reject the supplied stream at the canonical lexical-error location.
- **Alternatives considered:** Validate only monotonic spans and non-whitespace gaps; duplicate selected lexeme checks inside the parser; document that only tokenizer-produced vectors are supported; or remove the token-vector parser overload. Partial span or lexeme checks can drift from tokenizer rules and leave metadata forgery paths, documentation cannot constrain constructible public data, and removing the overload would unnecessarily break the staged API required by Issue #5.
- **Rationale:** Exact comparison against the existing tokenizer fixes the entire correspondence class with one source of lexical truth and no second scanner or token representation. Keeping the preliminary structural pass preserves safe handling and useful diagnostics for overflow-sized spans and malformed enum tags before comparison accesses public ranges.
- **Consequences or follow-up:** Public parsing now performs a second linear lexical pass and temporary token-vector allocation; this is the explicit cost of retaining an untrusted token-vector contract without credentials or hidden provenance. The normal source evaluator remains semantically unchanged. Any future tokenizer field or lexical rule automatically becomes part of the parser correspondence contract and must keep tokenizer and parser tests in sync.
- **Validation/evidence:** Before implementation, focused regressions reproduced all four rejected classes: omitted non-whitespace source, overlapping spans, out-of-order spans, and a forged `inc` kind over the lexeme `xxxxx`. The focused evaluator test reported all four failures against the integrated implementation, then passed after exact correspondence validation was added. Full Release, strict no-exceptions/warnings, sanitizer/adversarial, hygiene, and integrated CI gates remain required before handoff.

### 2026-07-18 — Use dependency-gated implementation and verification cards

- **Related issue:** [Issue #11](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/11)
- **Context:** Bennu's issue-branch workflow integrated Coder changes into `main` and then blocked the same Kanban card as `review-required`. Although terminal events notified Telegram, an outgoing notification did not dispatch Chef; independent verification therefore waited for a user reply or the three-hour stewardship poll. Issue #6 and later implementation work needs an immediate native handoff without weakening independent acceptance or the shared-main serialization rule.
- **Decision:** Represent each implementation issue from Issue #6 onward as `Implement #<number> (coder) -> Verify #<number> (chef)`, with the Chef card parent-gated on the Coder card. Coder completes only the implementation stage after the issue branch is validated, merged to local `main`, pushed, green on required integrated-main CI, and documented in a durable GitHub handoff while the issue remains open. Chef then independently verifies the merged range, acceptance criteria, runtime behavior, adjacent and error cases, and CI. Chef alone closes the issue and completes verification after recording evidence. A failed verification keeps or reopens the issue and creates a focused Coder correction card followed by a dependency-gated Chef re-verification card; failed work is never marked accepted. Keep one active Coder integration at a time, and permit concurrent Chef verification only in an isolated clean checkout that cannot disturb shared state. Retain Telegram terminal-event subscriptions and the three-hour stewardship cron only as notification and reconciliation safeguards.
- **Alternatives considered:** Keep blocking a single card for review; rely on Telegram replies or shorten the stewardship cron; add a custom task-event daemon; let Coder mark the issue accepted after CI; or allow Chef to inspect the shared checkout while another Coder integrates. These options were rejected because they preserve manual or polling latency, add operational machinery instead of using native dependencies, collapse implementation and acceptance, or risk cross-worker corruption and verification of moving state.
- **Rationale:** A parent edge is the smallest durable trigger the dispatcher already understands: implementation completion makes exactly one verifier ready immediately. Separate cards preserve role ownership and auditable state, while correction and re-verification stages represent failure honestly. Requiring integrated-main CI before Coder completion gives Chef a stable candidate, and checkout isolation preserves both serialization and independent evidence.
- **Consequences or follow-up:** This replaces the legacy single-card `review-required` convention for Issue #6 and later implementation work after Issue #11 is independently accepted. Orchestration must create both cards with the correct assignees and parent edge before dispatch and must avoid duplicate runnable implementation cards. Chef will validate the first Issue #6 graph after accepting this transition. Existing subscriptions and stewardship continue to detect or reconcile missed terminal events but are not part of the normal handoff path. This issue changes no Bennu product, build, test, release, daemon, cron, or Level 2 behavior.
- **Validation/evidence:** The workflow text is checked directly against every Issue #11 acceptance criterion, including implementation completion gates, open-issue ownership, independent verification evidence, failure routing, shared-main serialization, reviewer isolation, and safeguard roles. Repository validation and required integrated-main CI remain required before handoff; the first Issue #6 graph is intentionally deferred until independent Chef acceptance of Issue #11.

### 2026-07-18 — Terminalize rejected verification attempts without accepting product work

- **Related issue:** [Issue #11](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/11)
- **Context:** The initial two-stage workflow required a failed Chef verifier to remain incomplete while also requiring a parent-gated Coder correction. Those requirements deadlock the native dependency graph: the correction cannot become ready until its failed-verifier parent is terminal, but the verifier was forbidden from completing. The contract also needs to distinguish completion of a rejected verification attempt from acceptance of the product work.
- **Decision:** On verification failure, preserve or reopen the GitHub issue and record exact rejection evidence. Create the retry graph with explicit `failed verifier -> Coder correction -> Chef re-verification` parent edges. After both downstream task IDs exist, complete the failed verification-attempt card with structured metadata containing `accepted: false` and both task IDs. Treat that completion only as terminalization of the rejected attempt; only a later successful Chef verifier may record product acceptance and close the GitHub issue.
- **Alternatives considered:** Leave the failed verifier blocked and create an unparented correction; make the correction depend only on the original implementation; reuse the failed verifier for another run; or complete it without structured rejection metadata. These options were rejected because they deadlock the approved graph, bypass the failed-verifier handoff, blur distinct verification attempts, or make a terminal rejected attempt indistinguishable from acceptance.
- **Rationale:** Explicit parent edges preserve native dispatcher sequencing, while `accepted: false` records the outcome honestly and lets the failed attempt become terminal without claiming success. Separate correction and re-verification cards keep implementation and independent acceptance owned by the correct roles and make every retry auditable.
- **Consequences or follow-up:** Failed verification cards are terminal records, not successful product stages. Their metadata must identify the two retry cards, the GitHub issue remains open, and acceptance belongs exclusively to a later successful Chef verifier. The same correction-stage integration gates and re-verification evidence requirements continue to apply; no product, build, CI, daemon, cron, or Level 2 behavior changes.
- **Validation/evidence:** Chef independently reproduced the deadlock in the initial contract after integrated Main CI and Release CTest passed, and recorded the gap at [Issue #11 verification evidence](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/11#issuecomment-5012779523). Validation for this correction checks the exact three-stage retry edges, structured `accepted: false` completion, later-verifier-only acceptance, append-only diary history, repository tests, and integrated three-platform Main CI.

### 2026-07-19 — Reuse one evaluator path for recoverable REPL and atomic file execution

- **Related issue:** [Issue #6](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/6)
- **Context:** Bennu's accepted Level 1 evaluator returned structured values and located errors but had no user-facing execution path. The first CLI journey needs exact prompts and value prefixes, independent recovery after each bad interactive line, complete-file failure without partial stdout, portable source loading, and diagnostics that remain useful without C++ exception output.
- **Decision:** Link the existing `bennu_core` into the executable and keep CLI behavior as plain data plus free functions in `src/main.cpp`. The REPL flushes `> ` before each read, ignores horizontal-whitespace-only lines, evaluates each other line independently with `evaluate_expression`, emits successful values as `>>` plus `format_value`, and reports a failed line before continuing. A prompt already written before EOF remains the final stdout text and EOF returns status 0. The file runner reads the named path in binary mode through a fixed-size C stdio buffer and an explicit result record, then calls `evaluate_source` once and writes formatted values only after the complete result succeeds. Diagnostics use `source:line:column: category: message`; file-open and file-read failures use the same concise located file-error form.
- **Alternatives considered:** Build a second CLI parser; keep state across REPL lines; evaluate a file line by line and stream each early success; load files with `std::istreambuf_iterator`; print diagnostics on stdout; or add a CLI framework. These were rejected because they duplicate semantics, introduce Level 2 state, permit partial batch output, can surface an implementation-specific `std::__ios_failure` dump for directory/read failures, break machine-stable result streams, or add dependencies and abstractions for two commands.
- **Rationale:** One evaluator and formatter path keeps language behavior identical in library, REPL, and file use. Per-line REPL calls naturally isolate errors, while one complete-file call preserves all-or-nothing stdout. Fixed-buffer stdio makes open, read, and close failures visible in ordinary control flow and avoids the observed exception-based stream failure without adding a dependency.
- **Consequences or follow-up:** `emit-c`, `build`, persistent REPL context, history, completion, color, meta-commands, stdin batch execution, and later language levels remain unimplemented. Windows text-mode stdout may encode logical newlines as CRLF; source files are read byte-for-byte so the evaluator's existing LF/CRLF rules remain authoritative. Future source encoding or file-size policies require a separate decision.
- **Validation/evidence:** Test-first process cases observed the reserved-command failures before the canonical runner and REPL were implemented, missing diagnostic categories before recovery diagnostics were completed, blank-line errors before blank lines were ignored, argument-contract failures before exact dispatch was added, and file-error mismatches before explicit file loading replaced stream iteration. A directory reproduction previously aborted with `std::__ios_failure`; the explicit loader returns a concise nonzero file error instead. The Debug build's full CTest suite passes 22 tests, including real-executable transcripts, recovery, EOF, golden files, paths with spaces, LF/CRLF and blank lines, file failures, every source-reachable evaluator error class, and zero partial batch stdout. Clean Release, strict, sanitizer, and integrated CI gates remain required before handoff.

### 2026-07-19 — Emit validated Level 1 results as compact self-contained C11

- **Related issue:** [Issue #7](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/7)
- **Context:** Bennu needs a deterministic portable-C boundary for every accepted Level 1 program without adding a second grammar, a runtime dependency, or C compiler invocation. Emission must preserve the evaluator's signed 64-bit, ordering, overflow, type, formatting, and cumulative one-million-element allocation semantics, while an output failure must never truncate an existing destination. The maximum safe `ioata` output is several megabytes, so embedding a fully formatted output string would make generated source scale with result size.
- **Decision:** Add one free-function C emitter that calls the accepted `evaluate_source` pipeline before producing any C bytes. Use validated constant folding: emit each scalar root as an exact `INT64_C` expression (with a portable two-term expression for `INT64_MIN`) and each array root as its validated element count. Generate only the C11 standard headers and formatting helper functions needed by that program; array output is a bounded loop, so generated source stays compact and performs no dynamic allocation. Preserve root order in `main`, use `PRId64` for portable formatting, and return nonzero if generated-program stdout fails. Keep C compiler discovery and invocation exclusively in CTest, never in the `bennu` executable. Write output to an exclusively created same-directory sibling, close it successfully, then atomically replace the destination with POSIX `rename` or Windows `MoveFileEx` replacement; remove the sibling on every failure.
- **Alternatives considered:** Reparse source in a C backend; lower the public flat nodes into a second runtime evaluator; emit one giant escaped output string; write the destination directly; remove an existing Windows destination before a portable rename; or bundle/invoke a compiler. A second parser or runtime evaluator can drift from accepted semantics, giant strings scale source size with `ioata`, direct writes corrupt sentinels on late failure, remove-then-rename creates a data-loss window, and compiler invocation belongs to Issue #8.
- **Rationale:** Validated constant folding is the smallest complete backend for a side-effect-free Level 1 language. It makes every source/type/range/allocation error occur before output replacement, while compact scalar/count data and fixed helpers make the artifact inspectable, deterministic, standard-library-only, and independent of Bennu at runtime. Same-directory exclusive temporary creation keeps replacement on one filesystem and prevents collisions from overwriting unrelated files.
- **Consequences or follow-up:** Emission performs the evaluator's existing allocations before translation, including up to the documented cumulative one-million-element limit, but emitted C allocates no array storage and remains under a small fixed size plus one call per source result. Generated C is an executable implementation detail rather than a stable C API or optimized representation. Narrow command-line path handling remains consistent with the accepted runner; broader Unicode-path policy would require separate work. `bennu build`, compiler discovery/invocation, an external runtime ABI, optimization, packaging, and later language levels remain deferred.
- **Validation/evidence:** Test-first cycles observed the canonical `emit-c` journey fail as unimplemented, a direct destination write corrupt a sentinel in a non-writable directory, and empty-program C fail strict compilation because of unused helpers; each focused case passed after its corresponding implementation. Automated coverage inspects a committed golden C file, compiles and executes generated C11, compares direct and generated output through the maximum safe nested `ioata`, verifies signed boundaries/empty arrays/order/deterministic bytes/path independence, and exercises argument, input, semantic, output, collision, and atomic-preservation failures. Clean Release, strict warnings/conversions/Werror/no-exceptions, and ASan+UBSan builds each pass all 27 CTest cases. A detached canonical executable built with GNU C 15.2.0 runs after both C and temporary source files are removed and emits exactly `>>(1 2 3 4 5)` then `>>6`; duplicate emission hashes match and an invalid-source sentinel remains byte-identical with no orphan. Integrated three-platform CI remains required before handoff.

### 2026-07-19 — Invoke selected C compilers directly and publish native output last

- **Related issue:** [Issue #8](https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/8)
- **Context:** Portable C emission exists, but the one-command native path must select an installed toolchain predictably, pass hostile path and environment text without shell interpretation, support GCC/Clang-style drivers and the Windows release `cl.exe` driver, and preserve an existing executable across source, discovery, process, compiler, temporary, and output failures.
- **Decision:** Add a plain-data native-build component with free functions for compiler selection, target command-line construction, direct process launch, and output publication. Selection is exact: nonempty `--cc`, then nonempty `CC`, then `cc` fallback discovery on Linux/macOS or `cl.exe` on Windows. Explicit and environment selections are final and are never retried through fallback. Pass GCC/Clang `-std=c11 <source> -o <output>` arguments or MSVC `/nologo /std:c11 <source> /Fe:<output>` arguments as an argument vector. On POSIX, use `fork` plus `execvp`; on Windows, use `CreateProcess` with Microsoft-compatible argument quoting. Redirect compiler output to a bounded diagnostic log without involving a command shell. Emit through the accepted evaluator-backed C emitter, compile only inside an exclusively created same-filesystem temporary directory, verify the compiler's zero status and regular output file, move the candidate out, clean the isolated directory, and only then atomically replace the requested destination.
- **Alternatives considered:** Invoke `system`, a shell, or a command string; parse flags from `CC`; silently try fallback after an explicit selection fails; use one GCC-style command on every platform; compile directly to the requested destination; or compile under a global temporary directory. These were rejected because they enable interpretation or ambiguous boundaries, expand the Level 1 interface, hide configuration errors, fail the native MSVC contract, corrupt or misrepresent failed outputs, or prevent same-filesystem atomic publication.
