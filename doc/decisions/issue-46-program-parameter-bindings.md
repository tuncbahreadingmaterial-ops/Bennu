## 2026-07-23 — Bind borrowed ordered scalar values after whole-program lowering

- **Context:** BENNU-SPEC-0005 defines one optional leading `parameters[...]` program header, bare parameter references, program-wide static analysis before input handling, and typed scalar binding before execution. Issue #45 already established value-independent typed lowering and parameter-slot metadata. Issue #46 must expose this capability through the public evaluator while preserving the existing zero-input API, plain-data ownership, source diagnostics, and transactional results. CLI text decoding and generated/native argument parsing are assigned to Issues #47 and #48.
- **Decision:** Parse the header into a flat ordered declaration vector carrying name, declared scalar type, declaration/name spans, and header provenance. Parse bare names as either declaration-indexed parameter-reference nodes or unresolved-name nodes; syntactic bracket and prefix forms remain primitive calls. Resolve duplicate/reserved declarations and unknown names in source order, then complete existing whole-program arity/type/shape lowering before validating inputs. Add `evaluate_source` overloads accepting a borrowed `std::span<const Value>` with and without an explicit `EvaluationConfiguration`. Validate exact count first, then each ordered public value for container tag, scalar type tag, complete public `Value` invariants, scalar container, and exact declared type. Copy only the active scalar payload into evaluator-owned node slots, preserving Double bits. Return structured `ArgumentErrorContext`, `ParameterErrorContext`, and source spans in the public `Error`; keep the caller's input borrowed and never destroy or retain it. Reject parameter headers explicitly on expression and C-emission surfaces that do not yet accept parameter inputs.
- **Alternatives considered:** Use a name-to-value map; add text arguments to the evaluator; store borrowed `Value` references in runtime nodes; bind before lowering; represent each declaration/reference with objects; or extend generated/native argument handling in this issue. A map adds name allocation and lookup while the accepted contract is ordered. Text input belongs to Issue #47. Retained references complicate lifetime and ownership. Early binding violates static-error precedence and makes lowering value-dependent. Object graphs add indirection without capability. Generated/native parsing belongs to Issue #48 and is explicitly rejected rather than allowed to miscompile.
- **Rationale:** Declaration indices and `std::span` make data flow, ownership, and cost explicit. One linear validation pass and one scalar copy per reference reuse the existing flat postorder evaluator without allocation for bound scalar values or overload redispatch. Separating unresolved bare names from syntactic calls implements the accepted name-resolution semantics without weakening call syntax. Keeping static analysis before input validation preserves deterministic phase precedence and allows zero-root and unused-parameter programs to validate coherently.
- **Consequences or follow-up:** Repeated references produce independent evaluator-owned scalar slots from one borrowed binding; unused declarations remain required and validated. Existing programs without a header continue through the no-binding overload. The reserved `parameters` spelling is also rejected by primitive-table validation. REPL/evaluate-expression bindings remain unsupported, and parameterized C emission returns a structured unsupported-surface error until Issue #48 provides artifact argument handling. Issues #47 and #48 remain responsible for text decoding, `argc`/`argv`, process diagnostics, and generated/native parameter transport.
- **Validation/evidence:** `PARG-001-HEADER`, `PARG-001-NAMES`, `PARG-001-RESERVED-PRIMITIVE-DESCRIPTOR`, `PARG-002-STATIC-ORDER`, `PARG-003-SHAPE-ANALYSIS`, `PARG-004-TYPED-API`, `PARG-004-PARAMETER-LOWERING-METADATA`, `PARG-005-ARGUMENT-ERROR`, `PARG-008-ZERO-ROOTS`, `PARG-009-DYNAMIC-IOTA-SHAPE`, `PARG-010-RUNTIME-ORDER`, `PARG-011-PROFILES`, and `PARG-017-REGRESSION` cover malformed LF/CRLF/multiline headers and spans, declaration/name rules, program-wide static precedence, declaration-indexed lowering provenance, Bool/Int/Double binding and bit preservation, borrowed ownership, repeated/unused/zero-root behavior, `iota`, promotion, static/dynamic shape, overflow, resource refusal, multiple-root transactional order, every typed-value failure class, profile accounting, and program-only rejection. Focused Release tests passed during implementation; complete build, CTest, strict, sanitizer, public-header, traceability, and diff-hygiene evidence will be recorded in the pull-request handoff.

## 2026-07-23 — Exercise the program-only header rejection through the REPL

- **Context:** Issue #46 requires both the direct `evaluate_expression` boundary and the REPL user journey to reject a parameter header. The evaluator conformance test covered the direct API, but the process-level REPL route also needs durable cross-platform evidence.
- **Decision:** Add `cli.repl_rejects_parameter_header` to drive one complete parameter header through `bennu repl` and assert the existing program-only `SyntaxError` diagnostic, empty value output, continued prompt, and successful REPL process lifetime.
- **Alternatives considered:** Rely only on `PARG-017-REGRESSION`, or add parameter binding support to the REPL. The direct API test does not exercise CLI routing and presentation. REPL bindings are a non-goal and would contradict the accepted program-only boundary.
- **Rationale:** A focused process contract proves the real user surface delegates to `evaluate_expression` and rejects the header without accidentally treating it as a bindable program or terminating the interactive session.
- **Consequences or follow-up:** The REPL remains unparameterized. Issue #47 remains responsible for command-line text decoding on the program runner; this test only fixes the Issue #46 rejection boundary.
- **Validation/evidence:** `cli.repl_rejects_parameter_header` and `PARG-017-REGRESSION` cover the process and direct API paths respectively.
- **Supersedes:** None
