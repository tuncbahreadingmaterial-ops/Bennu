# Issue #49 — Implement structural tuple types, values, ownership, and resource accounting

- **Issue:** https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/49
- **File status:** Active append-only decision record

## 2026-07-23 — Use owned postorder arenas and one central profile-v2 tuple reservation boundary

- **Context:** BENNU-SPEC-0006 requires the tuple foundation before tuple source syntax exists. The implementation must preserve the established direct scalar/vector `Value` API, give diagnostics independently owned structural types, validate caller-formed records before access or transfer, move nested payload ownership exactly once, release arbitrary representable nesting without recursive host-stack use, and add backend-independent tuple-table accounting without changing frozen v1 behavior. Tuple-aware parser, evaluator, generated-C, native, spreading, and provenance behavior belong to later issues.
- **Decision:** Add an owned `TypeArena` with postorder nodes, contiguous child-index ranges, a final root, structural equality, validation, and iterative exact formatting. Generalize error type contexts to own complete arenas. Keep scalar/vector roots in the existing direct `Value` fields; for a tuple root, use the top-level `Value` as the logical final root and adopt every descendant into postorder `ValueNode`, child-index, vector-payload, and tuple-reservation tables. This preserves source compatibility while exposing the same checked single-root tree and independently owned payload identities. Construction validates every child, reserves the outer immediate table, then moves children in order without recharging adopted vector or nested tuple reservations. Public validation and queries are iterative and return stable invariant, tuple path, node, edge, and access-error context. Resource-aware release first validates, threads parent indexes temporarily into already-validated inactive node fields, then performs a linear allocation-free reverse-element depth-first pass before resetting the owner. Add exact `trusted-local-v2` and `bounded-v2` profiles, a fourth optional `max_tuple_table_bytes` limit, one fixed 16-byte canonical charge per immediate element, zero tuple work, a logical zero-byte empty reservation, and one shared positive-allocation ordinal sequence across vector, tuple, and workspace boundaries. V1 tuple construction fails with structured `ProfileError(unsupported_value_kind)` before reservation or ownership transfer.
- **Alternatives considered:** Force tuples into the former container-plus-scalar-element pair; replace every scalar/vector value with an arena immediately; store recursive owning child objects; borrow diagnostic types from runtime or analysis storage; clone nested payloads into outer tuples; charge host `sizeof(Value)` or transitive payload bytes; add tuple behavior to v1; recursively validate, format, or release; or retain an allocating cleanup worklist. The former type pair cannot express ordered heterogeneous structure, a wholesale public-value cutover creates unnecessary compatibility risk, recursive or borrowed ownership obscures lifetime and stack safety, cloning and transitive charging duplicate ownership/accounting, host sizes are backend dependent, mutating v1 breaks profile identity, recursion makes valid depth platform dependent, and a cleanup worklist can fail after ownership is already committed.
- **Rationale:** Postorder tables make tag, range, edge, parentage, payload, and reservation invariants locally inspectable while keeping ownership flat. A logical tuple root in the existing top-level record avoids an unrelated scalar/vector ABI redesign while retaining one final-root index and one owner path. Central reservation and release operations make canonical bytes, limit order, failure ordinals, and charge lifetime explicit. Temporarily repurposing inactive fields only after successful validation gives reverse depth-first cleanup linear cost with no recursion, allocation, extra persistent parent table, or public representation growth.
- **Consequences or follow-up:** Public `ErrorValueType` is now a complete owned `TypeArena`; callers that inspected the former container/element pair must inspect or format the structural arena. Tuple values are uniquely movable and not implicitly copyable because their payload handles use unique ownership. Host bookkeeping used to hold node and edge vectors remains outside profile accounting, while semantic tuple-table storage is charged only by the fixed 16-byte unit. Issue #50 still owns tuple grammar and evaluator/generated/native construction, #51 owns one-level static spreading and provenance, and later tuple-producing work must use this same reservation boundary. A future clone API, tuple-aware primitive signature, persistent parent table, or physical arena coalescing requires separate scope and must preserve canonical charges and fault ordinals.
- **Validation/evidence:** Focused public conformance exercises every structural type invariant, tuple/root/edge/payload/reservation failures, nested path propagation, empty/singleton/heterogeneous/deep formatting, owned diagnostic types, arity-first direct primitive rejection, unique move/release, exact and refused charges, v1 refusal, shared and nested allocation-failure ordinals, and scalar/vector regression. The focused suite, specification traceability, strict warning/no-exception build, full Release suite, sanitizer suite, diff hygiene, and exact-head CI evidence are recorded in the Issue #49 pull-request handoff.
- **Supersedes:** None. This implements the contract selected by `doc/decisions/issue-43-structural-tuples-profile-v2.md`, heading `## 2026-07-22 — Define tuples, static spreading, flat ownership, and profile v2 as one contract`.
