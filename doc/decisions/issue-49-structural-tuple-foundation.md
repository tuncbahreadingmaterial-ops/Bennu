# Issue #49 — Implement structural tuple types, values, ownership, and resource accounting

- **Issue:** https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/49
- **File status:** Active append-only decision record

## 2026-07-23 — Use owned postorder arenas and one central profile-v2 tuple reservation boundary

- **Context:** BENNU-SPEC-0006 requires the tuple foundation before tuple source syntax exists. The implementation must preserve the established direct scalar/vector `Value` API, give diagnostics independently owned structural types, validate caller-formed records before access or transfer, move nested payload ownership exactly once, release arbitrary representable nesting without recursive host-stack use, and add backend-independent tuple-table accounting without changing frozen v1 behavior. Tuple-aware parser, evaluator, generated-C, native, spreading, and provenance behavior belong to later issues.
- **Decision:** Add an owned `TypeArena` with postorder nodes, contiguous child-index ranges, a final root, structural equality, validation, and iterative exact formatting. Generalize error type contexts to own complete arenas. Keep scalar/vector roots in the existing direct `Value` fields; for a tuple root, use the top-level `Value` as the logical final root and adopt every descendant into postorder `ValueNode`, child-index, vector-payload, and tuple-reservation tables. This preserves source compatibility while exposing the same checked single-root tree and independently owned payload identities. Construction validates every child, reserves the outer immediate table, then moves children in order without recharging adopted vector or nested tuple reservations. Public validation and queries are iterative and return stable invariant, tuple path, node, edge, and access-error context. Resource-aware release first validates, threads parent indexes temporarily into already-validated inactive node fields, then performs a linear allocation-free reverse-element depth-first pass before resetting the owner. Add exact `trusted-local-v2` and `bounded-v2` profiles, a fourth optional `max_tuple_table_bytes` limit, one fixed 16-byte canonical charge per immediate element, zero tuple work, a logical zero-byte empty reservation, and one shared positive-allocation ordinal sequence across vector, tuple, and workspace boundaries. V1 tuple construction fails with structured `ProfileError(unsupported_value_kind)` before reservation or ownership transfer.
- **Alternatives considered:** Force tuples into the former container-plus-scalar-element pair; replace every scalar/vector value with an arena immediately; store recursive owning child objects; borrow diagnostic types from runtime or analysis storage; clone nested payloads into outer tuples; charge host `sizeof(Value)` or transitive payload bytes; add tuple behavior to v1; recursively validate, format, or release; or retain an allocating cleanup worklist. The former type pair cannot express ordered heterogeneous structure, a wholesale public-value cutover creates unnecessary compatibility risk, recursive or borrowed ownership obscures lifetime and stack safety, cloning and transitive charging duplicate ownership/accounting, host sizes are backend dependent, mutating v1 breaks profile identity, recursion makes valid depth platform dependent, and a cleanup worklist can fail after ownership is already committed.
- **Rationale:** Postorder tables make tag, range, edge, parentage, payload, and reservation invariants locally inspectable while keeping ownership flat. A logical tuple root in the existing top-level record avoids an unrelated scalar/vector ABI redesign while retaining one final-root index and one owner path. Central reservation and release operations make canonical bytes, limit order, failure ordinals, and charge lifetime explicit. Temporarily repurposing inactive fields only after successful validation gives reverse depth-first cleanup linear cost with no recursion, allocation, extra persistent parent table, or public representation growth.
- **Consequences or follow-up:** Public `ErrorValueType` is now a complete owned `TypeArena`; callers that inspected the former container/element pair must inspect or format the structural arena. Tuple values are uniquely movable and not implicitly copyable because their payload handles use unique ownership. Host bookkeeping used to hold node and edge vectors remains outside profile accounting, while semantic tuple-table storage is charged only by the fixed 16-byte unit. Issue #50 still owns tuple grammar and evaluator/generated/native construction, #51 owns one-level static spreading and provenance, and later tuple-producing work must use this same reservation boundary. A future clone API, tuple-aware primitive signature, persistent parent table, or physical arena coalescing requires separate scope and must preserve canonical charges and fault ordinals.
- **Validation/evidence:** Focused public conformance exercises every structural type invariant, tuple/root/edge/payload/reservation failures, nested path propagation, empty/singleton/heterogeneous/deep formatting, owned diagnostic types, arity-first direct primitive rejection, unique move/release, exact and refused charges, v1 refusal, shared and nested allocation-failure ordinals, and scalar/vector regression. The focused suite, specification traceability, strict warning/no-exception build, full Release suite, sanitizer suite, diff hygiene, and exact-head CI evidence are recorded in the Issue #49 pull-request handoff.
- **Supersedes:** None. This implements the contract selected by `doc/decisions/issue-43-structural-tuples-profile-v2.md`, heading `## 2026-07-22 — Define tuples, static spreading, flat ownership, and profile v2 as one contract`.

## 2026-07-23 — Make host failure and accounting ownership explicit at tuple boundaries

- **Context:** Independent review of the first Issue #49 pull-request head found seven conformance gaps hidden by the original passing suite: host allocation failures in type/value validation, equality, and formatting were not returned explicitly; tuple host metadata could fail before the injected semantic reservation; accounting could be released through an unrelated resource context; destruction did not refund active vector and tuple reservations by itself; duplicate physical vector buffers or tuple reservation handles could appear under distinct logical entries; root winner order inspected inactive tuple fields before final-root validity; and the shared reservation ordinal could wrap. These defects violated BENNU-SPEC-0006 Sections 4.3, 5.3, 5.4, 5.5, 6.3, 6.5, and 7.2 even though ordinary allocation and valid-value paths passed.
- **Decision:** Add explicit host-allocation failure results and deterministic host-allocation injection to structural type construction, validation, equality, type formatting, value validation, and value formatting. Preserve cheap structural winner checks before value-validation bookkeeping allocation. Establish the charged tuple reservation before fallible host metadata publication, and roll the reservation back without claiming children when metadata publication fails. Bind every active vector and tuple reservation to a shared live-byte accounting identity owned by its `EvaluationResources` context; copies and moves of that context retain the identity while independently created contexts do not. Resource-aware release rejects a different identity without mutation, while `destroy_value` releases all active transitive reservations directly through their stored identity before resetting the value. Validation rejects duplicate physical vector buffers and tuple reservation handles, including cross-kind aliasing. Root bounds and final-root identity win before inactive tuple-root payload checks. Positive reservation attempts reject an exhausted ordinal instead of wrapping it.
- **Alternatives considered:** Continue relying on throwing STL allocation and process termination; treat host bookkeeping as outside structured resource behavior; reserve all host metadata before the tuple allocation seam; use only an `accounting_active` bit; accept any caller-supplied resource context during release; retain raw pointers into `EvaluationResources`; make destruction storage-only and require a separate refund call; detect duplicate logical indexes without checking physical identities; or permit ordinal wrap because it is practically distant. Those choices make failure behavior configuration-dependent, leave uninjectable publication gaps, permit wrong-context refund or leaks after resource objects move, split exactly-once destruction across callers, admit double ownership, obscure deterministic invariant evidence, or violate monotonic allocation evidence.
- **Rationale:** A shared accounting identity is the smallest representation that remains stable when evaluator result records copy or move their resource context and allows values to refund their own active charges without an arbitrary context parameter. The tuple reservation followed by rollback gives one visible transaction boundary for semantic and host metadata failure while retaining children until commit. Explicit resource status keeps allocation unavailability distinct from malformed structural data, and ordinal preflight preserves deterministic evidence at the representational boundary.
- **Consequences or follow-up:** `EvaluationResources` now carries a shared live-byte accounting identity and is copy/move constructible but not assignable because its public live-byte reference must remain bound to that identity. `VectorValue` and `TupleTableReservation` retain shared accounting identities while active, so their accounting state remains valid after the originating resource record is moved or destroyed. The structural host-failure overloads are public fault-injection seams; ordinary overloads retain existing call syntax. Application validation translates host resource failure to `ResourceError` instead of inventing `InvalidValue`. Tuple metadata failure consumes the tuple reservation ordinal, restores live bytes, leaves children claimed, and reports that same ordinal.
- **Validation/evidence:** `TUP-003-VALUES`, `TUP-004-MOVE-CLEANUP`, `TUP-008-ALLOCATION-ORDINAL`, and `TUP-009-HOST-ALLOCATION-FAILURES` cover root winner order, inactive metadata, duplicate physical payload ownership, context-bound release, destruction refunds, moved accounting identity, ordinal exhaustion, host-allocation result propagation, and tuple metadata rollback. Final focused, Release, strict/no-exceptions, sanitizer, traceability, diff-hygiene, and exact-head CI results are recorded in the corrected pull-request handoff.
- **Superseded file and heading:** This corrects the transaction-order and cleanup assumptions in this file under `## 2026-07-23 — Use owned postorder arenas and one central profile-v2 tuple reservation boundary`; the owned flat representation and canonical accounting policy remain unchanged.

## 2026-07-23 — Complete failure-preserving queries, cleanup, and physical-owner validation

- **Context:** Final public-API review of the seven-finding correction found four boundary cases not covered by the first correction pass. Constructing an owned type through `value_type` could still fail after value validation without an injectable result; public destroy and reservation-release callers could not deterministically exercise validation exhaustion; malformed direct-vector release reset and freed the caller's record after validation failure; and physical-owner validation compared vector buffers only with other vector buffers, not tuple reservation storage or distinct tuple reservation owners. A moved-from `EvaluationResources` record also lost the shared pointer that kept its public accounting reference alive.
- **Decision:** Extend the host-allocation injection stream through owned `value_type` construction, public destruction, and resource-aware release, with resource failures kept distinct from invalid-value and conversion errors. Validate direct vectors before resource-context selection and leave every malformed owner and charge unchanged on failure. Require physical storage addresses to be unique across visited vector payloads and tuple reservations, including vector/tuple cross-kind aliases, before dereferencing vector elements. Preserve the accounting shared pointer in both source and destination when move-constructing `EvaluationResources`. Distinguish container-capacity overflow from allocation unavailability when reserving structural type bookkeeping.
- **Alternatives considered:** Treat post-validation type construction as infallible; test only validation and infer cleanup behavior; normalize malformed vectors during failed release; check only logical handle indexes or same-kind physical aliases; move the accounting shared pointer and declare the source unusable; or collapse every reserve failure to allocation unavailability. Those choices leave unreported fallibility, violate the caller-retains-input rule, retain double-free paths, permit a dangling public accounting reference, or erase the required overflow distinction.
- **Rationale:** One caller-supplied host-failure stream gives deterministic winner and preservation evidence across validation and the operation that follows it. Checking physical ownership before payload inspection prevents an alias from reaching a dereference or deallocation path. Copying a small shared pointer during a move is the simplest way to retain stable accounting identity and a valid source record without adding another ownership model.
- **Consequences or follow-up:** The additional overloads are fault-injection seams; ordinary call syntax and successful scalar/vector behavior remain unchanged. Failed public cleanup leaves ownership with the caller, which must repair or otherwise safely dispose of its caller-formed malformed record. Physical alias checks add comparisons only to public tuple validation and do not alter valid-value storage or release order.
- **Validation/evidence:** Fresh final validation passed the complete Release suite (54/54), strict `-Wall -Wextra -Wpedantic -Wconversion -Wsign-conversion -Werror -fno-exceptions` applicable suite (52/52), and ASan/UBSan/no-exceptions selected suite (3/3). Focused `TUP-*` passed 11/11 cases and 10,768 assertions, including injected value-type, destruction, release, malformed-preservation, same-kind/cross-kind owner-alias, context-identity, and moved-context checks.

## 2026-07-23 — Use one non-throwing host-storage boundary for structural operations

- **Context:** The deterministic host-allocation injection seams did not establish behavior when the real allocator refused memory. On the pre-correction Issue #49 baseline, an isolated valid 1,048,576-node type under a child-process `RLIMIT_AS` limit reached `std::vector::reserve`; the exception-enabled build caught `std::bad_alloc` and returned `allocation_unavailable`, while the required `-fno-exceptions` build terminated with exit 134. The same throwing-storage dependency existed in adjacent structural type/value validation, equality, construction, formatting, cleanup validation, and tuple metadata publication paths. The first malloc-backed correction also exposed two review obligations: an unchecked fixed-capacity push could write beyond its allocation if a capacity proof was wrong, and the direct-return scalar/vector type constructors could not report malloc failure without disguising it as an invalid empty semantic type.
- **Decision:** Store Issue #49 structural nodes, child indexes, value payload/reservation metadata, traversal/path scratch, and formatted output in `HostArray`, a fixed-capacity plain record backed by `malloc` and released by a free-function deleter. Allocation preflight returns `size_overflow` before consuming a synthetic allocation ordinal; a real null allocation returns `allocation_unavailable`; exception-enabled and no-exception builds execute the same code. Store the allocation capacity in the private allocation header as well as the public record, make push/fill return a checked `HostResourceErrorReason`, and require production callers to consume that status or prove aggregate capacity before ownership transfer. Size type-validation traversal by `child_indexes.size + 1`, a bound that remains sufficient before malformed graphs are rejected. Represent scalar and vector type leaves allocation-free in `TypeArena::inline_leaf`, expose their uniform node view through the free `type_nodes` functions, and retain malloc-backed postorder nodes for tuple arenas. Keep formatted text, validation paths, and dynamic primitive names in owning `HostArray` fields with `string_view` or `span` projections; profile views refer only to static execution-profile names.
- **Alternatives considered:** Retain throwing standard containers with catches only in exception-enabled builds; re-enable exceptions; install `new_handler`, termination, or process-global allocation hooks; treat virtual-address overcommit as allocation success; patch only the original type-validation probe; let an unchecked push rely on every caller remaining correct; return an empty invalid arena when leaf allocation fails; change the established leaf constructors to a new result API throughout unrelated diagnostic assembly; or force a direct Linux `libgcc_s` dependency with `--no-as-needed`. Those choices make supported configurations diverge, replace explicit recoverable errors with process behavior, leave adjacent abort paths or memory corruption possible, hide resource failure as semantic invalidity, expand the API correction unnecessarily, or couple this source correction to an unrelated packaging artifact.
- **Rationale:** One malloc-backed boundary gives allocator refusal a configuration-independent, non-throwing result and keeps ownership/allocation visible. Header-authoritative capacity plus checked writes makes a mistaken or corrupted public capacity claim fail without an out-of-bounds construction. The edge-derived traversal bound is valid without depending on alias, range, or orphan checks winning first. Inline leaves preserve the small coherent direct-constructor API without any fallible allocation, while tuple shapes still use the explicit-result construction contract. Owned backing arrays let result records move without dangling their diagnostic views because moving the owning pointer does not relocate its allocation.
- **Consequences or follow-up:** `TypeArena` now has two explicit public storage forms: scalar/vector leaves use `inline_leaf`, and tuple arenas use `nodes`; callers must use `type_nodes` when they need a uniform node sequence. Structural formatting and path-bearing result records, and errors with dynamic primitive names, are move-owned because their views reference embedded `HostArray` ownership. `ResourceErrorContext::profile` is a non-owning view only because every producer uses the static text returned by `execution_profile_name`. Host bookkeeping remains outside semantic profile accounting. The canonical Linux artifact tests continue to own their direct-library contract; the structural correction does not add linker forcing for a noncanonical `-fno-exceptions` executable.
- **Validation/evidence:** A Linux-only child-process test builds the structural sources with `-fno-exceptions`, applies `RLIMIT_AS`, and proves real allocator refusal returns explicit failure for type validation/construction/equality/formatting, value validation/type/formatting/destruction/release/detach, and tuple metadata publication without a signal or abort. Focused strict Release execution passed `unit.doctest` and `unit.structural_host_allocation_refusal`. Regression assertions cover a full array and a corrupted public capacity without overwriting the stored element, a malformed type with more edges than nodes, exact synthetic overflow ordinal preservation, and moved formatting/path/primitive-name ownership. ASan/UBSan with exceptions disabled passed the focused capacity and owned-error-lifetime cases.
