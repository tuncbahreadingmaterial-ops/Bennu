# Issue #64 — Adopt latest-base tree-identity verification for unchanged Level 3 PR patches

- **File status:** Active append-only decision record

## 2026-07-22 — Verify unchanged patches on a synthetic latest-base integration

- **Context:** Level 3 permits independent implementation and PR QA to overlap while serializing integration into protected `main`. When another accepted issue advances `main`, strict up-to-date protection makes a later branch stale. Repeating complete product QA after every conflict-free mechanical branch update is redundant, but accepting an old-base result or relying only on conflict absence can miss semantic interactions between disjoint files.
- **Decision:** Keep the previously reviewed patch head `H` unchanged while a verifier records current accepted `main` as `M` and constructs a clean synthetic integration `S = merge(M, H)` in a unique checkout without mutating the PR branch. A conflict routes correction or refresh plus fresh full independent QA with no reused product evidence. A clean `S` receives the full latest-base build, tests, acceptance journeys, scope checks, and relevant adjacent, error, resource, generated, and native validation. Only after that passes is the branch mechanically updated to `H2`; the update must add no product edits beyond integrating `M`, preserve the issue patch, and satisfy exact `tree(H2) == tree(S)`. Required source-bound CI and the App-owned Verity acceptance remain bound to exact `H2`. Immediately before guarded squash integration, Verity rechecks exact head and live `main == M`; afterward it proves merged-tree identity, waits for exact-merged-SHA push CI, and reruns focused post-merge acceptance before issue closure and success routing.
- **Alternatives considered:** Repeat complete independent product QA on every updated branch head; weaken or disable strict up-to-date protection; accept based on a clean merge, disjoint changed files, old-base CI, or old QA; or adopt a native GitHub merge queue now. Unconditional repeated QA wastes work on a byte-identical tree, weakening protection or accepting conflict absence loses current-base evidence, and a merge queue is a broader workflow change reserved for Level 4.
- **Rationale:** Full QA on `S` tests the exact latest-base product tree before branch mutation. Proving the later strict-protection update is integration-only and tree-identical transfers that current-base evidence to `H2` without claiming that old-base evidence remains valid. Exact-head, live-base, check-source, and post-merge guards preserve the existing independent acceptance boundary.
- **Consequences or follow-up:** `required_status_checks.strict` stays enabled, and the source-bound `PR Gate`, App-owned `Verity Meridian / QA accepted`, conversation resolution, administrator enforcement, squash-only integration, delayed issue closure, and success-only routing remain mandatory. Any conflict, manual resolution, changed patch, base movement, head movement, tree mismatch, or stale/mismatched check fails closed and requires the applicable correction or fresh latest-base verification. Consider a native merge queue only during Level 4 workflow planning.
- **Validation/evidence:** Review the complete Markdown-only diff and append-only record; run `git diff --check` and focused documentation/traceability checks; exercise the existing protected PR pipeline on the exact docs-only head; independently read back strict source-bound protection; and during integration record exact `H`, `M`, `S`, `H2`, tree identities, exact-head checks, guarded squash result, merged-tree identity, exact merged-SHA push CI, and focused post-merge acceptance.
