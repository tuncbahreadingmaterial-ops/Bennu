# Issue #45 — Build value-independent typed lowering with dynamic vector cardinality

- **Issue:** https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/45
- **File status:** Active append-only decision record

## 2026-07-23 — Lower once from structural types before execution

- **Context:** Level 2 coupled C emission to concrete evaluation. That made `emit-c` and `build` reject statically valid programs when literal values predicted a domain, resource, allocation, or dynamic-shape failure. BENNU-SPEC-0005 requires a whole-program static boundary with program-wide arity, type/signature, and statically knowable shape phases; Issue #46 will add parameter declaration parsing and binding after this lowering foundation exists.
- **Decision:** Introduce one flat typed lowering over the existing postorder arenas. Each lowered node records its scalar element type, `StaticScalar`/`StaticVector(n)`/`DynamicVector` cardinality, selected `PrimitiveImplementation`, argument slice, parameter-slot index when the node is a typed parameter reference, source location, and whether runtime shape agreement is required. The direct evaluator completes this lowering before creating values and executes the selected implementation without overload redispatch. C emission consumes the same lowering directly and never executes Bennu primitives. Generated C checks two completed vector lengths before result allocation or projection.
- **Alternatives considered:** Retain evaluator-backed emission with exceptions for known domain/resource failures; constant-fold literals to infer `iota` lengths; add a separate emitter-only type walk; or implement Issue #46's public parameter grammar and binding in this issue. The first three preserve duplicate or value-dependent semantics. The last would collapse the accepted dependency split and mix syntax/binding scope into the lowering foundation.
- **Rationale:** A single explicit lowering preserves BENNU-SPEC-0005's phase order and keeps overload/cardinality decisions independent of runtime values. Flat postorder vectors match the existing data-oriented representation, make ownership and traversal order visible, and let both evaluator and emitter consume identical typed decisions. Treating every `iota` result as dynamic avoids forbidden constant evaluation even for literal inputs.
- **Consequences or follow-up:** Predictable domain, profile, allocation, size, and dynamic-shape failures now allow `emit-c` and `build` to succeed and occur only when the artifact executes. Statically unequal literal-vector shapes still reject publication. Issue #46 must make the parser produce typed parameter-reference nodes with declaration/reference provenance and bind values into their indexed scalar slots; it does not need to redesign cardinality or primitive selection. Generated shape diagnostics remain the existing Level 2 process form until the later parameter/process-diagnostic work adds the full structured source-bound form.
- **Validation/evidence:** `typed lowering is value independent and retains dynamic shape data` covers typed Int/Double parameter-reference fixtures, `iota` dynamic cardinality, selected implementations, source provenance, and runtime-shape metadata. `typed lowering applies whole-program phase precedence` proves a later arity error wins over an earlier type error. `LOWERING-01 emission defers value-dependent failures to runtime` covers domain and dynamic-shape emission plus static shape/container rejection. `PUBLIC-ERROR-CLI-MATRIX` compiles and executes native domain, size, and dynamic-shape failures. Clean `Release` configure/build plus full CTest passed 53/53. A `-Wall -Wextra -Wpedantic -Werror -fno-exceptions` build passed; 51/53 tests also passed under those non-release link flags, while the two ELF/package policy tests rejected the intentionally changed direct `NEEDED` set (`libgcc_s.so.1` is not direct with `-fno-exceptions`), and both passed in the required Release run. ASan+UBSan build and the seven focused evaluator, differential C, resource, native, and traceability tests passed. Manual direct/generated-strict-C11/native journeys produced byte-identical valid output, emitted and deferred domain/dynamic-shape failures to artifact execution, and rejected static shape mismatch during emission. `git diff --check` passed.
- **Supersedes:** None

## 2026-07-23 — Preserve lowering shape anchors and artifact provenance at runtime

- **Context:** Independent implementation review found that lowering recorded a known static vector length but the evaluator discarded it before application. A dynamic vector before a static vector could therefore become the runtime anchor and attribute the mismatch to the static argument. Generated C also collapsed every dynamic shape failure to bare `ShapeMismatch`, despite BENNU-SPEC-0005 requiring the original logical source position and the same primitive, argument, and shape context as direct execution.
- **Decision:** Keep shape agreement driven by the flat lowering metadata. Before primitive allocation or kernel work, the evaluator checks every dynamic-vector argument against the lowering's static length when one exists; without a static length, the first dynamic vector remains the runtime anchor. C emission writes equivalent explicit checks before `bennu_apply`. Generated runtime failure state retains the primitive name, one-based mismatching argument position, expected and actual lengths, and the mismatching argument's source location, and serializes them with the embedded `bennu-source` label.
- **Alternatives considered:** Continue rediscovering shape from runtime `Value` order in `apply_typed_primitive`; reorder arguments so a static vector is visited first; or leave generated diagnostics compact for later parameter work. Rediscovery contradicts the lowering's authoritative cardinality, reordering changes observable argument precedence, and deferral violates the already accepted zero-parameter generated/native diagnostic contract.
- **Rationale:** The lowering already owns the value-independent proof and the static anchor. Consuming that data directly preserves argument order, runs the check before outer resource or kernel work, and keeps evaluator and emitted C behavior equivalent without a second type/shape dispatch system.
- **Consequences or follow-up:** Dynamic shape failure state uses the existing flat generated-resource record and adds no object graph or allocation. Generated code performs only the checks marked by lowering. Artifact shape diagnostics are now source-bound and contextual for existing zero-parameter programs; Issue #46 can reuse the same mechanism when public parameter parsing and binding arrive.
- **Validation/evidence:** `typed runtime shape checks honor static anchors and first mismatch order` covers dynamic/static, static/dynamic, and dynamic/dynamic order in the evaluator. `PARG-012-VALUE-INDEPENDENT-EMISSION` runs the same cases through the runner, strict-C11 emitted program, and native builder and requires exact logical source, primitive, argument, and expected/actual shape diagnostics with empty stdout. `typed lowering checks type errors before static shape errors across roots` covers type-over-shape whole-program traversal, and traceability maps each normative section to these focused checks.
- **Supersedes:** The generated-shape-diagnostic consequence in `2026-07-23 — Lower once from structural types before execution`.

## 2026-07-23 — Bind every generated dynamic failure to its semantic source context

- **Context:** Supplemental post-merge conformance review found that generated and native size, allocation, profile, and integer-domain failures did not satisfy BENNU-SPEC-0005 section 12.2. Size, allocation, and domain paths emitted unlocated compact strings, while profile refusal used a separate offset-bearing record. Those paths also failed to retain all of the direct evaluator's stable resource or domain context in the generated failure state. The differential test encoded the weaker artifact strings instead of rejecting them.
- **Decision:** Extend the flat generated resource state with the direct error model's dynamic context: requested element/byte presence and values for resource failures; profile limit, usage, refusal, and admission data; and selected implementation, operands, optional result index, primitive, and source position for integer overflow. Every dynamic resource/domain setter records the originating lowering admission point and source location before control reaches the common failure label. The artifact formatter validates the required context, prefixes every diagnostic with `bennu-source:<line>:<column>`, reproduces the direct domain/size/allocation message semantics, and preserves the full stable profile refusal fields. Shape serialization and publish-last zero-stdout behavior remain unchanged.
- **Alternatives considered:** Prefix only the existing compact strings; derive location from the last executed node at the common failure label; or expose a new process-only structured record. Prefixing would leave kind context incomplete, a last-node heuristic would misattribute child/postorder failures, and a new record would diverge from the accepted section 12.2 presentation contract. Capturing context at the failure site is direct and preserves the evaluator's ownership and failure order.
- **Rationale:** Dynamic failures are created where requested sizes, selected implementations, operands, result index, profile accounting, primitive identity, and source location are simultaneously known. Recording those plain fields there avoids reconstruction, allocation, or a parallel object model. The common failure state gives emitted C and native builds byte-identical source-bound diagnostics while retaining enough data to enforce the same kind, reason, and structured context as direct execution.
- **Consequences or follow-up:** Generated/native size overflow, allocation unavailable, profile limit, scalar integer overflow, and lifted integer overflow now identify the logical Bennu source position and primitive. The generated runtime rejects incomplete dynamic context as an internal failure rather than publishing a misleading semantic diagnostic. This correction adds no parameter parsing, `argc` handling, argument count/decoding, or other Issue #46 behavior.
- **Validation/evidence:** `LOWERING-ARTIFACT-DYNAMIC-DIAGNOSTICS` derives each domain/size/shape artifact diagnostic from the runner's exact category/message and replaces only the physical source label, then checks strict-C11 emitted and native status, stderr, and empty stdout for scalar/lifted domain failure, size failure, child-before-parent order, shape order, and cross-root precedence. `PARG-012-RESOURCE-DIAGNOSTICS` verifies exact direct resource context and source positions, then executes emitted/native profile and injected-allocation failures repeatedly with source-bound diagnostics, resource cleanup, and atomic output. Traceability narrows the earlier section 11 claims to the covered dynamic sub-order and does not claim Issue #46 argument behavior.
- **Supersedes:** The remaining generated domain/resource diagnostic consequence in `2026-07-23 — Lower once from structural types before execution`; it preserves the shape correction above.
