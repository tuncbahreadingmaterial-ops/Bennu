# Issue #55 — Add checked arithmetic primitives

- **Issue:** https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/55
- **File status:** Active append-only decision record

## 2026-07-23 — Extend one checked scalar-kernel model across five arithmetic primitives

- **Context:** Bennu exposed checked `inc` and `add`, but lacked decrement, numeric negation, absolute value, subtraction, and multiplication. The new operations must preserve Int64 data integrity without C++ signed-overflow undefined behavior, use one stable domain error model, retain section 16 scalar lifting and `Int -> Double` promotion, and produce identical evaluator, emitted-C, and native behavior. Double results must be binary64 round-to-nearest-ties-to-even independently of the ambient host rounding mode, with signed zero, infinities, subnormals, and canonical NaN preserved as specified.
- **Decision:** Add stable `dec`, `neg`, `abs`, `sub`, and `mul` primitive identities, each with exact `Int` and `Double` signatures, and assign explicit numeric values to every flat implementation ID shared with generated C11. Int kernels prove bounds before performing fixed-width arithmetic and report the existing `integer_overflow` reason with primitive, signature, operands, and optional lifted result index. Double `dec`, `sub`, and `mul` execute one binary64 operation under a saved strict floating-point control environment; `neg` complements the sign bit and `abs` clears it; every result passes through the existing canonical value boundary. Typed lowering, scalar broadcasting, vector agreement, promotion, resource accounting, error precedence, and publish-last behavior remain shared rather than operation-specific. The generated C11 runtime mirrors the same flat implementation IDs, checks, strict floating-point control, and structured dynamic failure state. C++ arithmetic translation units and every emitted/native C compile path explicitly disable contraction and fast math, enable rounding-mode access, and use MSVC strict floating-point mode where applicable.
- **Alternatives considered:** Implement Int arithmetic directly and inspect the result afterward; use compiler overflow intrinsics; implement `dec` as `add[x -1]`, `neg` as `sub[0 x]`, or `abs` as conditional syntax; rely on the process floating-point environment; or add operation-specific overflow reasons. Post-hoc signed arithmetic invokes undefined behavior, compiler intrinsics do not provide the same portable C++20/C11 surface on all supported targets, composition would add unavailable language machinery and obscure primitive identity, ambient rounding is externally mutable, and distinct reasons duplicate information already carried by the primitive and signature.
- **Rationale:** Precondition checks make every potentially overflowing operation locally inspectable and permit the arithmetic expression only after safety is established. Reusing the existing flat primitive descriptor, implementation enum, application path, and generated failure record keeps one source of lifting and diagnostic semantics. Saving and restoring the x64 MXCSR or arm64 FPCR/FPSR around each arithmetic operation gives deterministic binary64 behavior while preserving caller state and avoiding recoverable exceptions or new dependencies.
- **Consequences or follow-up:** The public primitive set grows by five names and ten scalar overloads. Programs may now express checked integer arithmetic and IEEE binary64 arithmetic through the evaluator, emitted standard C11, and native builder. Each strict Double arithmetic call has a small control-register save/restore cost; `neg` and `abs` remain exact bit transformations. Other arithmetic such as division remains outside this issue.
- **Validation/evidence:** Focused scalar-kernel tests enumerate Int64 boundaries, exchanged multiplication operands, ordered subtraction cases, vector-safe no-partial-result behavior, binary64 signed-zero/subnormal/finite-overflow/infinity/NaN vectors, rounding-sensitive cases under every directed rounding mode, and exact host rounding/sticky-flag restoration. Section 16 matrix tests enumerate all nine elementwise primitives. The tracked public success corpus and dynamic error matrix compare direct evaluation, strict-C11 emitted execution, and native execution, including lifted overflow indices, hostile floating-point control state, and rounding-sensitive Double results. Exact command results and cross-platform PR checks are recorded in the Issue #55 pull-request handoff.
- **Supersedes:** None

## 2026-07-23 — Isolate generated Double formatting from ambient floating-point control

- **Context:** The arm64 PR journey showed that generated arithmetic restored the caller's hostile flush-to-zero and directed-rounding state correctly, but generated decimal formatting then inherited that state. On macOS arm64, `snprintf`/`strtod` could not round-trip the smallest subnormal under flush-to-zero and reported an output failure even though the computed binary64 value was correct.
- **Decision:** Reuse the generated runtime's saved strict floating-point environment around each complete Double-capable value-formatting call, then restore the raw caller control and status before continuing. Keep arithmetic kernels independently guarded so their semantics remain correct outside the generated program's publication loop.
- **Alternatives considered:** We considered weakening the hostile-environment test, keeping strict mode active for the whole generated execution, or replacing decimal conversion. Weakening the test would hide a cross-path semantic difference, whole-execution strict mode would enlarge the restoration surface across allocation and error paths, and replacing conversion is unnecessary when the existing formatter is correct under the specified binary64 mode.
- **Rationale:** The formatting call is the smallest complete region containing every floating conversion and round-trip check. A single entry/restore pair there keeps output deterministic, preserves the caller's exact environment even on output failure, and avoids duplicating architecture-specific register logic.
- **Consequences or follow-up:** Generated and native output now remains byte-identical under hostile rounding and flush-to-zero state on x64 and arm64. Each published Double-containing root pays one additional environment save/restore pair; no new allocation, dependency, or language-visible state is introduced.
- **Validation/evidence:** `cli.emit_c_differential` runs a hostile x64/arm64 harness and verifies both exact corpus bytes and raw control restoration. The macOS arm64 exact-head PR matrix provides the native arm64 regression evidence; Linux x64 Release, strict, and sanitizer journeys cover the same generated path locally.
- **Supersedes:** None

## 2026-07-23 — Require representation-level and same-process backend evidence

- **Context:** Independent QA of PR #76 found that the implementation's focused evaluator tests inspected binary64 bits, but the tracked cross-backend row cited formatted stdout as if it proved interchange-bit identity. The generated/native structured probe covered only legacy `inc`/`add`, hostile floating-point state was installed only around one emitted executable, and public resource/error fixtures did not individually bind every new checked primitive to allocation, work, cleanup, and deterministic-failure assertions.
- **Decision:** Treat formatted output, evaluator values, generated-runtime state, and native-runtime state as separate observables. Keep the public formatted corpus for runner/REPL/emitted/native presentation parity, but narrow its traceability claim. Extend the generated runtime's internal same-process probe to inspect raw binary64 bits for `dec`, `neg`, `abs`, `sub`, and `mul`; run that identical probe through both strict-C11 compilation and Bennu's native-builder route; install directed rounding plus flush/denormal controls; enable an overflow trap while executing new `mul`; and compare the complete supported control/status state after every call. Give the generated structured failure record an explicit stable domain-reason enum instead of deriving `integer_overflow` only while formatting stderr. The same probe now checks every new Int implementation's stable identity, reason, signature, typed operands, scalar/lifted index semantics, source spans, no-partial-result state, allocator balance, deterministic first failure, result-allocation refusal, work refusal, zero-work empty execution, and cleanup. Direct evaluator and public-path fixtures retain their own structured and lifting assertions.
- **Alternatives considered:** Continue relying on shortest-roundtrip text, add a public bit-inspection language primitive, launch each failure in a fresh process, or duplicate one generated program per operation. Text cannot expose NaN normalization or signed-zero representation reliably enough for this contract, a public inspection primitive would expand the language solely for tests, fresh processes mask stale state and leaks, and duplicated programs would multiply orchestration without improving the observed runtime seam.
- **Rationale:** The generated runtime already exposes a static execution/application seam inside emitted translation units. Compiling one allocator-instrumented probe by both supported routes observes the exact kernels and state records without adding production ABI. Separating traceability rows prevents a green formatting test from carrying a stronger claim than its assertions.
- **Consequences or follow-up:** Acceptance tests are larger and exercise architecture-specific x64/AArch64 control registers, but the production language and runtime ABI remain unchanged. Unsupported floating-point architectures still fail closed. Public success/error corpora now enumerate the new lifting surfaces, while exact-bit and hostile-environment claims belong only to the representation-level probe.
- **Validation/evidence:** `unit.doctest` owns evaluator raw-bit and structured-overflow assertions. `public.api_resource_matrix` compiles and executes the same generated probe as emitted strict C11 and through the native builder, twice in one process, with allocator accounting and hostile/trap state. `cli.emit_c_differential`, `cli.public_path_error_matrix`, `docs.smoke`, and `spec.traceability` bind formatted surfaces, dynamic failures, documentation, and exact evidence names independently. Exact command and platform-matrix results are recorded on PR #76.
- **Supersedes:** The validation-evidence scope in the first 2026-07-23 entry is narrowed: formatted cross-path output is presentation evidence, not raw binary64 evidence.

## 2026-07-23 — Isolate the complete supported x64 rounding environment

- **Context:** Fresh independent QA of PR #76 showed that the prior x64 hostile-environment probe changed MXCSR directly and therefore did not model a standard C caller. On Linux x64, `fesetround(FE_UPWARD)`, `FE_DOWNWARD`, or `FE_TOWARDZERO` changes both MXCSR and the x87 control word. The generated runtime saved and normalized only MXCSR, so `snprintf`/`strtod` inherited directed x87 rounding while formatting checked-Double results and failed to find a round-tripping representation. Each directed mode returned `OutputError` with no stdout even though arithmetic and MXCSR restoration were correct.
- **Decision:** On GNU/Clang x64, extend `BennuStrictEnvironment` to save the complete x87 environment with `fnstenv`, install masked round-to-nearest x87 control for the guarded operation, and restore the saved x87 environment with `fldenv`; continue saving, normalizing, and exactly restoring MXCSR independently. The supported MSVC x64 path retains its MXCSR guard because the x64 UCRT standard rounding control is represented there. Keep the existing arm64 FPCR/FPSR guard unchanged. Add same-translation-unit emitted and native probes that use standard `fesetround` for all three directed modes around `bennu_execute`, compare exact checked-Double output bytes, and verify the standard rounding/exception observables plus raw architecture control/status are unchanged after every call. GCC-like native builds link the standard math library so the production native-builder seam can compile the standard-C fenv probe.
- **Alternatives considered:** Use `feholdexcept`/`fesetenv` in every generated strict region; change only the decimal formatter; replace decimal conversion; or broaden the existing direct-MXCSR test without calling `fesetround`. Standard fenv library calls would add a runtime dependency and repeated libc calls to every guarded arithmetic operation, a formatter-only special case would leave the shared strict-environment abstraction incomplete, replacing conversion is unnecessary, and direct MXCSR mutation cannot reproduce the x87 state that caused the defect.
- **Rationale:** The defect is an incomplete machine-state boundary, not a decimal algorithm defect. Saving and restoring the architecture environment directly makes the data flow explicit: caller x87 and MXCSR state enter the guard, both execution domains run under masked nearest rounding, and both exact states are restored before control returns. The standard-C probes exercise the public caller contract that the register-only probe missed, while the retained direct probes continue covering DAZ/FTZ, sticky state, enabled traps, arm64 control/status, and raw result bits.
- **Consequences or follow-up:** GNU/Clang x64 strict regions use 28 additional stack bytes and one x87 environment save/restore pair; MSVC x64 and arm64 code generation are otherwise unchanged. GCC-like native compiler command lines add `-lm`; ordinary generated programs retain the same language behavior and output. Unsupported architectures still fail closed.
- **Validation/evidence:** Before the production change, the tracked standard-C emitted probe reproduced the QA failure: `cli.emit_c_differential` returned exit 1 with `OutputError: stdout failure` under the first directed mode. After the change, `cli.emit_c_differential`, the same-source native standard-fenv probe in `public.api_resource_matrix`, and focused `unit.doctest` validation pass. The probes cover `dec` at the largest finite value, halfway `sub`, rounding-sensitive `mul`, subnormal multiplication, and smallest-subnormal subtraction; each of FE_UPWARD, FE_DOWNWARD, and FE_TOWARDZERO produces the exact 60-byte corpus and restores caller mode, standard exception state, and raw architecture state.
- **Supersedes:** The x64 scope and validation claims in the first two 2026-07-23 entries are corrected: MXCSR-only isolation is not the complete standard C x64 floating-point environment.
