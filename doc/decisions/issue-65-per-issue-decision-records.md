# Issue #65 — Split the decision diary into append-only per-issue records

- **Issue:** https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/65
- **File status:** Active append-only decision record

## 2026-07-22 — Replace the shared diary tail with issue-owned records

- **Context:** The shared `doc/decision-diary.md` had grown to 305,327 bytes and 788 lines through accepted Issue #42 work. Every material decision appended to the same tail, and parallel Issues #39 and #42 produced a real merge conflict even though their product scopes were independent. The complete history must remain auditable without allowing future issues to recreate that write hotspot.
- **Decision:** Move the complete historical diary without changing its bytes to `doc/decisions/legacy-decision-diary.md`; leave `doc/decision-diary.md` as a small compatibility pointer; and record each issue's new decisions in at most one stable lowercase-ASCII `doc/decisions/issue-<number>-<kebab-slug>.md` file. Multiple decisions and same-issue corrections append new sections to that file. Cross-issue supersession is recorded only in the later issue's file. Discovery uses the issue number, filename search, and Git history rather than a tracked central index.
- **Alternatives considered:** Continue appending to one diary; split historical entries into per-issue files; maintain a generated or hand-edited central index; use an ADR framework or database; or allow one file per decision. The shared tail preserves the observed conflict class, splitting history would rewrite provenance, an index creates another shared write target, external machinery adds dependencies and operational surface, and one file per decision weakens issue ownership and increases naming choices.
- **Rationale:** One stable append-only file per issue isolates independent writes while preserving a direct issue-to-record mapping. An unchanged legacy blob retains historical evidence exactly, and a short old-path pointer keeps existing links discoverable without accepting new entries.
- **Consequences or follow-up:** Contributors and reviewers must follow `doc/decisions/README.md`, append only to the active issue's record, and avoid routine edits to shared policy, template, compatibility, legacy, or other issue files. Pending Issue #64 and later Level 3 work must create or append to their own issue records. Historical links may continue through the compatibility pointer, but tests or active contracts that inspect historical decision text must read the legacy file explicitly.
- **Validation/evidence:** At implementation base `5946ac8c117b93d422a3f4e417f0b41413279e9c`, `doc/decision-diary.md` had Git blob `44e01d3026ff2a43dfe0a784e4d4c60f869757fc` and SHA-256 `945649552bb022a11394438c389caf000f0b732ec457825e293c005a4cde6bff`. The migration validation compares the moved path byte-for-byte and by Git blob identity with that base object, inventories every tracked old-path reference, and requires documentation checks plus the complete Release and applicable strict suites.
- **Supersedes:** `legacy-decision-diary.md`, heading `2026-07-18 — Establish a minimal, normative contributor contract`, only for the shared single-diary storage decision; all other historical content remains in force unless separately superseded.
