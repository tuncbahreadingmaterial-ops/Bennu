# Issue #43 — Specify structural tuples, one-level prefix spreading, and profile-v2 accounting

- **Issue:** https://github.com/tuncbahreadingmaterial-ops/Bennu/issues/43
- **File status:** Active append-only decision record

## 2026-07-22 — Define tuples, static spreading, flat ownership, and profile v2 as one contract

- **Context:** Level 3 needs fixed-length heterogeneous values that compose with the existing right-associated primitive-name prefix grammar and later sequential fan-out. The former Level 2 grammar rejects `add [1 2]`; the current scalar/vector type pair cannot describe ordered heterogeneous structure; nested ownership must remain unique and safe at arbitrary representable depth; the frozen `trusted-local-v1` and `bounded-v1` profiles have no tuple-table unit; and evaluator, generated strict C11, and native execution need identical failure points without using host layout sizes. Issue #43 is specification-only and must leave tuple-aware primitives, recursive flattening, destructuring, callable values, mixed explicit spread, and fan-out syntax to separate work.
- **Decision:** Adopt `[]`, `[1]`, and `[1 2.5 true]` as immutable structural tuple expressions while preserving immediately adjacent `name[...]` as direct application and `(...)` as homogeneous vector syntax. A primitive-name prefix call still parses one complete right-associated operand; if that operand's static type is `Tuple`, call preparation expands exactly its immediate children before arity, signature/conversion, shape, resource, domain, and execution checks. Direct calls and public primitive application never spread; direct public application preserves arity-first checking and reports malformed caller-owned data as structured `ValueError(invalid_value)` before type selection. Represent both structural types and owned values as checked postorder node arenas plus contiguous child-index ranges and one root, with unique parentage, no aliases/cycles, and independently owned diagnostic type arenas. Owned values move uniquely; spreading borrows immutable child views; no implicit tuple copy or runtime reference counting exists. Canonical value formatting uses brackets and one ASCII space between elements, while public types use exact `Tuple<...>` notation. Runtime values contain no source spans; typed-program sidecars map each outer element to a literal child span, a construct-defined producer span, or the whole forwarding expression. Add `trusted-local-v2` and `bounded-v2`; bounded v2 may configure `max_vector_bytes`, new `max_tuple_table_bytes`, `max_live_evaluation_bytes`, and `max_work_units`, while trusted v2 omits all four. Charge each tuple's immediate table exactly once as `element_count * 16` canonical bytes, charge tuple construction zero work, preserve every nested vector/tuple reservation without a second charge, and assign zero-based fault ordinals only to admitted positive-size vector, tuple, or workspace allocation attempts. Empty tuples have a logical zero-byte reservation and no allocation ordinal. V1 profiles remain frozen and reject tuple programs before execution. Evaluate tuple elements once left to right, admit the outer table only after all succeed, transfer owners infallibly, clean failures and completed roots in deterministic reverse order, and require iterative or flat traversal for parsing, type operations, validation, formatting, and cleanup with no language depth cap.
- **Alternatives considered:** Treat whitespace before `[` as another direct-call spelling; spread tuples in every argument position; decide spreading from available overloads; recursively flatten; add an explicit spread marker now; use recursive object graphs, borrowed diagnostic type trees, raw forgeable element pointers, shared ownership, or reference counting; charge transitive payloads again through each outer tuple; derive charges from `sizeof(Value)`; mutate the v1 profiles; impose arbitrary tuple-arity or nesting caps; allocate the outer table before evaluating children; or permit recursive host-stack traversal. Alternate direct-call spelling would not create first-class values, global or overload-dependent spread makes existing programs unstable and removes tuple-preserving calls, recursive flattening destroys nested structure, and an explicit mixed marker is broader follow-up syntax. Recursive/borrowed/shared representations obscure ownership and make deep cleanup unsafe, while raw pointers cannot support safe public validation. Transitive or host-sized accounting double counts or becomes backend dependent. Mutating v1 breaks frozen profile identity, arbitrary caps lack a product basis, early admission changes failure order and live usage, and host recursion turns valid depth into a platform accident.
- **Rationale:** One grammar split at adjacency preserves existing direct applications while giving the formerly invalid form a composable meaning. Static one-level expansion keeps argument interpretation independent of runtime data and future overload additions. Flat postorder arenas make ownership, validation, formatting, and release inspectable and permit allocation-free iterative destruction. A fixed 16-byte semantic slot is simple, portable, and independent of C++ or C layout. Separate v2 names make the new observable charge and ordinal sequence explicit without weakening v1 compatibility. Producer sidecars preserve precise diagnostics through computed tuples without coupling reusable values to source buffers.
- **Consequences or follow-up:** Issue #49 must implement flat type/value ownership, validation, formatting, v2 profiles, and the central reservation boundary; #50 must implement tuple source/lowering/construction across evaluator, C, and native paths; #51 must implement static one-level prefix expansion and provenance; and #44/#53 must use the same v2 tuple-table event for fan-out results. Those issues must add executable traceability identifiers from BENNU-SPEC-0006's plan. A future tuple-aware signature, clone API, explicit mixed spread, destructuring, recursive flattening, callable value, or new resource kind requires a separate specification. Optimization may physically coalesce storage or elide an immediate tuple only if every canonical charge, allocation ordinal, failure, provenance, and lifetime observation remains identical.
- **Validation/evidence:** The normative contract is BENNU-SPEC-0006, with explicit amendments and links in BENNU-SPEC-0001, 0002, 0004, and 0005 and a resolved-status link from the exploratory Level 3 record. Its requirement-to-test table enumerates parser/type/value/ownership/profile/allocation/construction/spreading/provenance/output/deep-nesting/strict-C/native/platform evidence. This specification change is validated by repository documentation and traceability checks, a clean Release build and full CTest suite, applicable strict configurations, `git diff --check`, and the protected exact-head pull-request matrix before implementation handoff.
